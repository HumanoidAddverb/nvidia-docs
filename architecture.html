    Addverb Arms → Isaac Sim: Teleoperation & Data Collection Bridge¶

    This package uses the Addverb 6-DOF robotic arm as the physical leader in a sim-to-real teleoperation pipeline backed by Isaac Lab (Isaac Sim). The sim follower is mounted on a table top, a cube spawns in front of it, and the operator picks the cube up with the physical leader while RGB streams, joint positions and leader/follower commands are logged in the exact same Parquet + JPEG format used by addverb_vla_datacollection.

    The Addverb arm has 6 DOF. The driver reports a 7-element joint vector where indices [0:6] are arm joints and index [6] is the gripper (same unit: radians).

    Installation¶

    Step 1 — Isaac Lab (Sim PC only)¶

    Follow the official Isaac Lab installation guide: https://isaac-sim.github.io/IsaacLab/main/source/setup/installation/index.html

    This repo also ships a Docker image — see Docker (Sim PC) below for the supported path.

    Step 2 — Addverb pybind driver (Hardware PC)¶

    cd addverb_arms/python_bindings
# Follow the build instructions in that directory (cmake + pip install)
pip install -e .

    Step 3 — Python dependencies¶

    Both PCs: 

    pip install pyzmq numpy PyYAML click

    Hardware PC only — for the R/E keyboard listener in hardware_bridge_sender.py: 

    pip install pynput      # X11 session; stdin fallback otherwise

    Sim PC (in addition to Isaac Lab's own env): 

    pip install torch opencv-python scipy pandas pyarrow

    Or everything from the requirements file: 

    pip install -r addverb_isaaclab/requirements.txt

    Step 4 — Robot USD asset¶

    The robot is loaded from a pre-converted USD bundle that lives in the repo:

    addverb_isaaclab/assets/usd/addverb_arm/
    addverb_gripper_urdf_2.usd            # top-level (robot articulation, variants)
    configuration/
        addverb_gripper_urdf_2_base.usd   # geometry (~63 MB)
        addverb_gripper_urdf_2_physics.usd # joints + drives + welds + slide axis flip
        addverb_gripper_urdf_2_sensor.usd
        materials/

    run_sim.sh mounts assets/usd/ into the container read-write and the agent loads addverb_gripper_urdf_2.usd directly. There is no URDF→USD conversion step. To swap in a new robot, replace the bundle in place (the addverb_gripper_urdf_2.usd top-level uses relative refs into configuration/).

    Step 5 — Joint name verification¶

    Launch the env once and copy the names from the Isaac joint order: ... banner into config/agent_config.yaml → kinematics.isaac_joint_names.

    Where do task prompts come from?¶

    The text written into each parquet row's task_prompt field is resolved from dataset_config.yaml → task_ranges. Each entry is [start_episode, end_episode, task_name, task_prompt]; at log time SimDataLogger._get_task_prompt(episode_num) finds the entry whose [start, end] range contains the active episode and stamps that prompt into both the row and meta/tasks.parquet. Edit task_ranges to attach different prompts to different episode-number ranges.

    Domain randomisation¶

    isaac_scene_config.yaml → domain_randomization exposes per-knob enabled flags so each block can be turned on/off independently:

    domain_randomization:
  camera_ego_focal_length:
    enabled: false
    range_mm: [12.0, 15.0]
  camera_external_pose:
    enabled: false
    pos_range: { x: [-0.02, 0.02], y: [-0.02, 0.02], z: [-0.01, 0.01] }
    rot_range: { roll: [-0.05, 0.05], pitch: [-0.05, 0.05], yaw: [-0.05, 0.05] }
  light_exposure:
    enabled: false
    range_ev: [-3.0, 1.0]

    Each enabled block adds a corresponding EventTermCfg(mode="reset", ...) to AddverbEventCfg (via sim_agent/addverb_env_cfg.py), backed by the helper functions in sim_agent/mdp/resets.py (randomize_camera_focal_length, randomize_camera_pose, randomize_light_exposure). With every block disabled (the default) the env behaves identically to a no-DR run.

    Rendering / Latency Tuning¶

    Teleop latency is dominated by sensor rendering. All rendering knobs live in one section of isaac_scene_config.yaml:

    rendering:
  mode: "performance"                # "performance" | "balanced" | "quality"
  enable_translucency: false
  camera_update_period_s: 0.0333     # ≈ 30 Hz sensor update
  carb_settings: {}                  # raw carb overrides — use with caution

    mode: "performance" already turns off RTX reflections / AO / DLSS and picks cheap defaults for shadows. enable_translucency is a first-class Isaac Lab attribute and safe to flip.

    carb_settings is a raw pass-through — Isaac Lab validates keys against its known list and an unknown key aborts env construction (for example rtx.post.dlss.enabled is rejected on Isaac Lab 2.3). Only put a key in carb_settings after you've confirmed it exists on your Isaac Lab version.

    The container mounts addverb_isaaclab/config/ as read-write, so edits on the host take effect at the next container launch without a rebuild.

    If teleop feels sluggish, tune in this order: 1. Increase camera_update_period_s — biggest single win (30 Hz → 20 Hz → 15 Hz). 2. Drop camera_ego.width/height and camera_external.width/height (render cost scales quadratically). 3. Leave mode: "performance". 4. Raise physics.decimation (fewer renders per second at the cost of physics accuracy).

    Running¶

    Docker (recommended, Sim PC)¶

    From General_Autonomy_Special_Robots/:

    sg docker -c "./docker/run_sim.sh --task Addverb-Teleop-Cam-v0"

    Extra args after run_sim.sh are forwarded to addverb_agent.py — e.g. --leader_addr tcp://<hw-ip>:5556 or --repo_root /workspace/datasets/run1 --episode_num 0.

    The container host-mounts: - addverb_isaaclab/config/ → :rw (edit YAMLs on the host, relaunch container). - datasets/ → /workspace/datasets:rw (dataset output).

    Bare-metal (Hardware PC — physical arm)¶

    cd addverb_isaaclab

# Bring up the CAN interface first (if not already)
../addverb_arms/setup_all_cans.sh

# Canonical publisher
python bridge/bridge_sender.py --config config/bridge_config.yaml

# OR the keyboard-enabled variant (R = reset, S = save & home, E = exit)
python bridge/hardware_bridge_sender.py --interface can0 --model ADD

    Bare-metal (Sim PC — teleop only)¶

    python addverb_isaaclab/sim_agent/addverb_agent.py \
    --task Addverb-Teleop-Cam-v0 \
    --leader_addr tcp://<hardware-pc-ip>:5556

    Bare-metal (Sim PC — with data collection)¶

    python addverb_isaaclab/sim_agent/addverb_agent.py \
    --task Addverb-Teleop-Cam-v0 \
    --leader_addr tcp://<hardware-pc-ip>:5556 \
    --repo_root /data/addverb_sim_dataset \
    --episode_num 0

    Keyboard Controls (hardware_bridge_sender.py)¶

    When launched from a terminal with an X display (or a TTY fallback), the hardware bridge watches for two keys:

    Key Effect
    R Call arm.reset_to_home() on the physical arm and publish cmd:reset → sim env calls env.reset(), re-spawning the cube.
    E Publish cmd:exit → sim agent shuts down cleanly, bridge homes the arm and exits.

    Ctrl+C / SIGTERM are also handled as "exit" so you never leave the physical arm under power unexpectedly.

    Leader Calibration¶

    Run once (or whenever the leader's home drifts). Easiest path — the launcher reads launch_config.yaml + bridge_config.yaml, activates the conda env, and backs up any existing calibration.yaml:

    ./scripts/launch_calibration.sh
# overrides:
INTERFACE=can1 ./scripts/launch_calibration.sh
MODEL=ADD      ./scripts/launch_calibration.sh

    Or call the underlying script directly:

    python bridge/hardware_bridge_sender.py \
    --interface can0 --model ADD --calibrate

    The flow: 1. You move the leader to the sim home pose (all 6 arm joints centred), press ENTER. 2. The bridge averages 50 samples and computes joint_offset_rad. 3. Move the gripper to its fully-OPEN mechanical pose (the slider-crank zero), press ENTER → physical_open_rad. 4. Values are written to config/calibration.yaml.

    KinematicMapper automatically overlays calibration.yaml on top of agent_config.yaml the next time the sim agent starts ([KinematicMapper] overlaid calibration from ... log line). Hardware actuator angle theta_raw maps to the slide position via the slider-crank equation:

    theta   = theta_raw - physical_open_rad
x_mm    = 35 * (1.6128 - cos theta - sqrt(1.249 - sin^2 theta))
slide_m = clamp(x_mm / 1000, 0, 0.05)

    The same slide_m drives BOTH slide_1_joint and slide_2_joint each step (the env writes the targets independently — no mimic — and the USD authors slide_2's joint frame flipped 180° about X so equal positive joint values produce mirrored world motion = paddles separate).

    Hardware-Free Testing (Mock Mode)¶

    You can exercise the full simulation + logging pipeline without physical Addverb hardware using the mock publisher.

    # Terminal 1 — mock sender (static zeros or slow sine)
python bridge/mock_bridge_sender.py --sine

# Terminal 2 — sim agent
python addverb_isaaclab/sim_agent/addverb_agent.py \
    --task Addverb-Teleop-Cam-v0 \
    --leader_addr tcp://127.0.0.1:5556

    MDP (sim_agent/mdp/)¶

    The MDP layer is split into observation / reset / termination modules, following the Isaac Lab MDP convention:

    • obs.py — image, image_raw, ee_frame_state, cube_position.
    • resets.py — reset_joints_by_offset, random_asset_pose, reset_cube_on_table, plus the camera / light DR helpers.
    • terms.py — cube_grasped, cube_lifted, cube_lifted_termination (position-based detection — no ContactSensor required).

    terms.cube_lifted_termination is not wired into the default env. For teleop you drive episode boundaries with the bridge R key; enabling the termination would cause the sim follower to auto-reset mid-motion. Add it when you want headless / RL-style evaluation:

    from isaaclab.managers import TerminationTermCfg as DoneTerm
from .mdp import cube_lifted_termination

@configclass
class AddverbTerminationsCfg:
    cube_lifted = DoneTerm(
        func=cube_lifted_termination,
        params={"table_top_z": 0.4, "lift_height": 0.15, "confirm_steps": 25},
    )

    Edge-Case Handling (Agent + Bridge)¶

    • Malformed ZMQ frames — the receiver thread validates shape / finiteness and logs rate-limited parse errors rather than dying.
    • Stale leader link — if no frame arrives for > 1 s the agent holds the last command and warns; recovery is logged when frames resume.
    • NaN / non-finite joint values — caught in the bridge and the agent.
    • Control messages — the bridge publishes {"cmd": "reset"} and {"cmd": "exit"} on the same socket; the agent acts on them before applying any pending joint command.
    • Clean shutdown — SIGINT / SIGTERM latch an exit request; the bridge publishes cmd:exit, homes the arm, closes sockets.

    Collecting Data¶

    End-to-end smoke-tested at 30 Hz; output is fully compatible with addverb_vla_datacollection (same helper functions, same directory tree, same parquet schema). A single recording session of N episodes ends up as a single growing dataset — parquets are appended, info.json totals are kept in sync, and each episode gets its own episode_000/, episode_001/, ... subfolder under every camera.

    1. Confirm prerequisites¶

    • Hardware PC: bridge up and publishing, leader calibrated (--calibrate at least once — see Leader Calibration above).
    • Sim PC: the Docker image is built; datasets/ exists at the project root (the run script creates it on first launch).

    2. Launch the sim with logging enabled¶

    From General_Autonomy_Special_Robots/:

    # Bridge + sim on the same machine (the common case — default works):
sg docker -c "./docker/run_sim.sh \
    --task Addverb-Teleop-Cam-v0 \
    --repo_root /workspace/datasets/session_001 \
    --episode_num 0"

    If the bridge runs on a different host, set --leader_addr explicitly. Wrap the URL in quotes or avoid angle-bracket placeholders — an unquoted <hardware-pc-ip> is treated as input redirection by bash:

    sg docker -c "./docker/run_sim.sh \
    --task Addverb-Teleop-Cam-v0 \
    --leader_addr 'tcp://10.0.4.117:5556' \
    --repo_root /workspace/datasets/session_001 \
    --episode_num 0"
    • The default --leader_addr (tcp://localhost:5556, set in agent_config.yaml) works because run_sim.sh uses --network=host.
    • --repo_root must be inside /workspace/datasets/ (mounted read-write from General_Autonomy_Special_Robots/datasets/ on the host). The first episode creates the folder; later episodes append.
    • --episode_num starts at 0 and should be incremented by hand for each new recording (the logger does not auto-increment across agent restarts). See "Multi-episode recording" below for the pattern.
    • --task Addverb-Teleop-Cam-v0 (note: -Cam) is required for logging — Addverb-Teleop-v0 omits the cameras.

    3. Teleoperate¶

    • Two floating viewport windows (WristCam, TableCam) open next to the main Isaac Sim viewport. Whatever they show is exactly what the dataset is capturing.
    • Drive the Addverb leader to perform the task (pick up the red cube, for example). The agent maps leader joints → sim actions every physics step; the logger drains the most recent frame at 30 Hz and writes it out.
    • Press R on the keyboard that owns the bridge to reset both the physical arm and the sim env between attempts. The current episode's rows stay in memory; they are only flushed to parquet on finalize() (exit or cmd:exit).
    • Console shows a rate-limited log line: [SimDataLogger] frame=... arm=... gripper=....

    4. End the episode¶

    Press E on the bridge keyboard. The sim receives cmd:exit, calls logger.finalize(), and shuts down. finalize() appends the buffered rows to data/chunk-000/file-000.parquet, updates meta/episodes/.../file-000.parquet, and rewrites meta/info.json with the new total_episodes and total_frames.

    5. Multi-episode recording¶

    Increment --episode_num for each recording and keep --repo_root the same:

    # episode 0
./docker/run_sim.sh --task Addverb-Teleop-Cam-v0 \
    --repo_root /workspace/datasets/session_001 --episode_num 0

# episode 1 (after exiting the previous session)
./docker/run_sim.sh --task Addverb-Teleop-Cam-v0 \
    --repo_root /workspace/datasets/session_001 --episode_num 1

    Each run produces one episode. Stale --episode_num values overwrite images into the same episode_XXX/ folder but add a new row to the episodes parquet — pick numbers you haven't used yet in that session.

    6. Inspect the result¶

    On the host (not inside the container):

    # Tree view
tree -L 4 datasets/session_001

# Row count + columns
python -c "
import pandas as pd
df = pd.read_parquet('datasets/session_001/data/chunk-000/file-000.parquet')
print('rows:', len(df))
print('columns:', list(df.columns))
print(df.head(3))
"

# Metadata
python -c "import json; print(json.dumps(json.load(open('datasets/session_001/meta/info.json')), indent=2))"

    Quick recording checklist¶

    • Leader calibrated (--calibrate recent, config/calibration.yaml present)
    • Bridge running on hardware PC (hardware_bridge_sender.py — keyboard focused)
    • Sim container launched with -Cam task, --leader_addr, --repo_root, --episode_num
    • WristCam + TableCam popup windows visible and updating
    • R resets both arms and respawns the cube
    • E flushes the episode and exits cleanly
    • Dataset folder has meta/, data/chunk-000/, and images/observation.images.{ego,external}/episode_XXX/
    • meta/info.json shows robot_type: addverb_sim_6dof, fps: 30.0, correct total_episodes / total_frames

    Dataset Format¶

    Output is fully compatible with addverb_vla_datacollection — the sim logger uses the exact same metadata_utils_parquet.py helpers. A dataset captured in sim drops straight into the same pipeline as real-hardware data. See the Output layout section in Data Collection (Sim) for the on-disk tree.

    Parquet row schema:

    Column Shape Description
    observation.state.joint_positions [7] Sim follower joint positions — [link_1..link_6 (rad), slide_1 (m)]. slide_2 mirrors -slide_1 every step and is intentionally not logged.
    action.robot [7] Commanded joint targets in the same layout — 6 arm rad + 1 gripper drive in metres
    state.base [2] Zeros (no mobile base)
    action.base [2] Zeros (no mobile base)
    base.timestamp_us int64 Leader hardware timestamp (µs)
    timestamp float32 Wall-clock time (seconds)
    frame_number int64 Monotonic frame counter

    Default logging rate is 30 Hz, matching addverb_vla_datacollection. Set dof.base_obs_dim: 2 / dof.base_act_dim: 2 in dataset_config.yaml for cross-compatibility with dual-arm mobile data — the logger zero-pads to [9]-element vectors.

    Troubleshooting¶

    Symptom Fix
    KeyError: 'joints' in receiver Bridge is publishing the old arm_pos / gripper_pos schema — update to the canonical {joints, ee_pose} payload (already done in hardware_bridge_sender.py).
    assert actual_dof == n_dof fails Set bridge_config.yaml → robot.n_dof to match the arm's reported DOF
