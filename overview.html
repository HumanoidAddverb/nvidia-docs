    QDD Arms + Isaac Lab — Hybrid Development Environment¶

    Hardware on host. Simulation in Docker. Connected by ZMQ.

    Documentation map¶

    This README is the Overview page of the docs site. The full set of operator guides is published at docs.local:

    Page What it covers
    Overview This file — architecture, prerequisites, quick start, validation, project layout, configuration reference, troubleshooting.
    Architecture & Setup Detailed addverb_isaaclab/ package layout, scene/USD/camera config, full pipeline run-through.
    Isaac Lab Notes Hybrid host + container architecture, GPU/X11 passthrough, Docker build internals.
    Data Collection (Sim) Hardware leader → Isaac Sim follower; calibration, 3-terminal flow, Parquet+JPEG output, multi-episode.

    If you are new, read this page, then jump to whichever page matches your task — calibration & teleop go to Data Collection (Sim), physical-arm captures go to Real-Hardware Data Collection, and training/eval flows live in Sim-to-Real Training.

    Full sim-to-real pipeline at a glance¶

    The repository covers the complete sim-to-real loop for the 7-DoF QDD arm (6 active revolute joints + 1 welded wrist flange in sim + 1 prismatic gripper). Five stages, each documented in detail on its own page:

    Domain-randomisation ranges (camera ego focal length, camera external pose, light exposure) live in addverb_isaaclab/config/isaac_scene_config.yaml → domain_randomization. Each block has its own enabled: true|false flag (defaults are false), and the helper functions in addverb_isaaclab/sim_agent/mdp/resets.py are wired into reset events by addverb_isaaclab/sim_agent/addverb_env_cfg.py only for blocks that are enabled. Toggle a single knob without rebuilding the env. Cube spawn position is always randomised at every reset (driven by isaac_scene_config.yaml → cube.spawn_range).

    The dataset format is identical between sim and real captures, which is what makes Stage 5 co-training straightforward — point the trainer at a directory containing both Parquet roots and let the sampler do the mixing.

    Prerequisites¶

    Requirement Version
    OS Ubuntu ≥ 22.04
    GPU NVIDIA RTX (Ada / Blackwell recommended)
    NVIDIA Driver ≥ 525 (your system: 580.126.09)
    Docker ≥ 24.0
    NVIDIA Container Toolkit latest
    Conda (Miniconda / Mambaforge) any

    Install NVIDIA Container Toolkit (if not installed)¶

    curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
    sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
    sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
sudo apt-get update && sudo apt-get install -y nvidia-container-toolkit
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker

    Quick Start¶

    Step 1 — Set up the HOST Conda environment¶

    cd /home/vla-training/isaac_vla/General_Autonomy_Special_Robots

# Create the conda environment (host-side only)
conda env create -f unified_env.yml
conda activate unified_addverb_isaaclab

    Step 2 — Build the QDD Arms C++ pybind module¶

    conda activate unified_addverb_isaaclab

cd addverb_arms/python_bindings/pybind
mkdir -p build && cd build
cmake ..
make -j$(nproc)
# Output: ../addverb_arms_interface.cpython-311-x86_64-linux-gnu.so

    Verify the build: 

    cd ../../..  # back to python_bindings/
conda run -n unified_addverb_isaaclab python test_import.py
    Expected output: 6 passed, 0 failed

    Step 3 — Build the Docker image¶

    cd /home/vla-training/isaac_vla/General_Autonomy_Special_Robots

chmod +x docker/build_sim.sh docker/run_sim.sh
./docker/build_sim.sh

    First build pulls ~20 GB from NVIDIA's registry. Subsequent builds are cached.

    Running the System¶

    Quickest path — single-config launchers¶

    Edit launch_config.yaml at the repo root once (mode: mock|real, image, task, repo_root, etc.), then:

    # Terminal A (host) — bridge
./scripts/launch_bridge.sh

# Terminal B — sim
./scripts/launch_sim.sh

    Per-run overrides: EPISODE_NUM=3 ./scripts/launch_sim.sh, MODE=real ./scripts/launch_bridge.sh. See scripts/launch_*.sh for the full list of env-var knobs. The sections below describe the manual pipeline if you want to invoke the bridge and container directly.

    Calibration¶

    ./scripts/launch_calibration.sh                  # all defaults from launch_config.yaml + bridge_config.yaml
INTERFACE=can1 ./scripts/launch_calibration.sh   # override CAN interface
MODEL=ADD      ./scripts/launch_calibration.sh

    The launcher runs the interactive arm + gripper calibration (joint home, gripper open, gripper closed, optional jaw separation in metres) and writes addverb_isaaclab/config/calibration.yaml. After the prompts the bridge falls into its publishing loop, so you can leave it running and start the sim straight away — press E or Ctrl+C if calibration was all you wanted.

    Terminal A — Host: Start the bridge sender¶

    Without hardware (mock sine wave for testing):¶

    conda activate unified_addverb_isaaclab
cd /home/vla-training/isaac_vla/General_Autonomy_Special_Robots

python addverb_isaaclab/bridge/mock_bridge_sender.py --sine

    With real hardware (CAN bus):¶

    conda activate unified_addverb_isaaclab
cd /home/vla-training/isaac_vla/General_Autonomy_Special_Robots

# Ensure CAN interface is up first:
# sudo ip link set can0 up type can bitrate 1000000
python addverb_isaaclab/bridge/bridge_sender.py \
    --config addverb_isaaclab/config/bridge_config.yaml

    Terminal B — Container: Launch Isaac Sim¶

    The robot is loaded from a pre-converted USD bundle committed under addverb_isaaclab/assets/usd/addverb_arm/addverb_gripper_urdf_2.usd (mounted into the container read-write). No URDF→USD conversion happens at launch. For more detail on the Isaac Lab side, see Isaac Lab Notes.

    cd /home/vla-training/isaac_vla/General_Autonomy_Special_Robots

# Basic teleop — no cameras, no recording
./docker/run_sim.sh --task Addverb-Teleop-v0

# With cameras
./docker/run_sim.sh --task Addverb-Teleop-Cam-v0

# With exactly matched hardware-style Data Recording (Parquet + JPEGs)
./docker/run_sim.sh --task Addverb-Teleop-Cam-v0 --repo_root /workspace/datasets/session_001 --episode_num 0

    Validation Test (No Hardware Required)¶

    Run this to confirm host ↔ container communication works end-to-end:

    Terminal A — start mock bridge¶

    conda activate unified_addverb_isaaclab
cd /home/vla-training/isaac_vla/General_Autonomy_Special_Robots
python addverb_isaaclab/bridge/mock_bridge_sender.py --sine

    Expected output: 

    [mock_bridge_sender] Publishing on tcp://0.0.0.0:5556 at 200 Hz
[mock_bridge_sender] Mode: Sine wave mock motion
[      0] arm: [0. 0. 0. 0. 0. 0.]  gripper: 0.0000 rad
[   1000] arm: [0.048 0.048 ...]     gripper: 0.0480 rad

    Terminal B — launch sim container¶

    ./docker/run_sim.sh --task Addverb-Teleop-v0

    Expected console output in the container: 

    [addverb_agent] Task:        Addverb-Teleop-v0
[addverb_agent] Leader addr: tcp://localhost:5556
[ZMQLeaderReceiver] Subscribed to tcp://localhost:5556
[addverb_agent] Running. Close the Isaac Sim window or Ctrl+C to stop.

    The Isaac Sim viewport will open and the robot arm will mirror the sine wave motion from the mock sender.

    Configuration Reference¶

    addverb_isaaclab/config/bridge_config.yaml¶

    Controls the host-side bridge sender.

    Key Default Description
    robot.can_interface can0 CAN bus device name
    robot.n_dof 7 Arm DOF (6 arm + 1 gripper)
    zmq.host 0.0.0.0 Bind address (all interfaces)
    zmq.port 5556 ZMQ PUB port
    control.loop_hz 200 Publishing rate

    addverb_isaaclab/config/agent_config.yaml¶

    Controls the container-side Isaac Lab agent.

    Key Default Description
    env.task Addverb-Teleop-v0 Gym environment ID
    env.device cuda:0 Torch compute device
    zmq.leader_addr tcp://localhost:5556 Bridge sender address
    logging.enabled false Dataset recording toggle
    logging.dataset_root /tmp/addverb_sim_dataset Dataset output path

    Dependency Conflict Notes¶

    Conflict Resolution
    numpy 2.x vs Isaac Sim 1.26.0 Host env uses <2.0; container base pins 1.26.0
    torch 2.5.x (cu124) vs Isaac Sim 2.7.0 Separate environments — no conflict
    opencv-python vs headless envs Use conda opencv package (no Qt) on host; pip headless in container
    lerobot datasets requirement Installed inside container only; host doesn't need it
    torchvision==0.20.1 vs lerobot >=0.21.0 Relaxed to >=0.20.1 on host; container uses base image version

    Troubleshooting¶

    Isaac Sim window doesn't appear¶

    xhost +local:docker
# Re-run docker/run_sim.sh

    ZMQ: agent shows joints is None (no data received)¶

    1. Confirm bridge_sender is running on the host
    2. Confirm port 5556 is not blocked: ss -tlnp | grep 5556
    3. With --network=host, localhost in the container == host machine ✓

    Docker can't access GPU¶

    nvidia-smi  # verify driver works
docker run --rm --gpus all nvidia/cuda:12.6.0-base-ubuntu22.04 nvidia-smi

    launch_bridge.sh aborts with CONDA_BUILD: unbound variable¶

    The conda activate.d hook ros-noetic-catkin_activate.sh references $CONDA_BUILD, $target_platform, $build_platform without defaults, and set -u in the launcher trips on them. The launcher already wraps the conda activate call with set +u / set -u; if you copy this script into a new project make sure that pair stays intact.

    CAN interface not found¶

    # Bring up CAN interface
sudo ip link set can0 up type can bitrate 1000000
# Or use the provided setup script:
bash addverb_arms/setup_slcan.sh
