    Real-Hardware Data Collection — Single Leader + Single Follower¶

    End-to-end recipe for capturing teleoperation episodes on physical hardware only (no Isaac Sim, no AMR base): one leader arm drives one follower arm, two USB cameras (top + wrist) stream synchronised RGB, and every episode lands in the same LeRobot-flavoured Parquet + JPEG layout used by the dual-arm pipeline. The follower arm has 6 active joints + a parallel-jaw gripper (link_7 is welded as a fixed flange in sim — irrelevant on hardware, where it has never been actuated).

    The whole pipeline runs inside a self-contained Docker container — the host only needs Docker and the SocketCAN devices wired up.

    Source path: addverb_arms/addverb_vla_datacollection/ Image: addverb-datacollect:latest (built from Dockerfile.datacollect in the same directory)

    1. What gets recorded¶

    Per frame, written at config_single.LOGGING_RATE_HZ (default 30 Hz):

    Field Type Shape Meaning
    index int64 (1,) Global frame index across all episodes
    episode_index int64 (1,) Episode number this row belongs to
    frame_index int64 (1,) Index within this episode
    task_index int64 (1,) Resolved from EPISODE_TASK_RANGES
    timestamp float32 (1,) Wall-clock seconds (epoch)
    observation.state.joint_positions float32 (7,) Follower joints j0..j5 + gripper
    action.robot float32 (7,) Leader joints j0..j5 + gripper (commanded)
    observation.images.camera1_images image (H,W,3) Top camera (default 320×320)
    observation.images.camera2_images image (H,W,3) Wrist-mounted camera

    The schema is identical to the dual-arm pipeline minus the second arm and the AMR base — driven entirely by config_single.py (NUM_ARMS=1, BASE_OBS_DIM=0, BASE_ACT_DIM=0, JOINT_DIM=7, ACTION_DIM=7).

    3. Files in this pipeline¶

    addverb_vla_datacollection/
├── config.py                ← dual-arm config (unchanged)
├── config_single.py         ← single-arm config (NEW)
├── data-master.py           ← orchestrator (patched, fully back-compat)
├── data-collect.py          ← recorder    (patched, --single-arm/--no-base flags)
├── logger_thread.py         ← logger      (patched, base_client now optional)
├── camera_thread.py         ← unchanged
├── camera_config.json       ← 3 USB cameras
├── metadata_utils_parquet.py← unchanged (parquet writer)
├── remote_transfer_client.py← unchanged (TCP streamer)
├── collect_data_single.bash ← single-arm launcher (NEW)
├── Dockerfile.datacollect   ← container image (NEW)
└── run_datacollect_docker.sh← container launcher (NEW)

    The single-arm flow does not duplicate the dual-arm code. It re-uses data-master.py, data-collect.py, logger_thread.py, etc. by:

    1. Adding --single-arm and --no-base flags to data-collect.py / data-master.py so the second arm pair and the AMR base client are simply not constructed.
    2. Making base_client truly optional inside DataLoggingThread — when None, the base-sync gate is skipped and state.base / action.base / base.timestamp_us columns are dropped from every row.
    3. Aliasing import config → config_single for the duration of one run via a mktemp-staged shim prepended to PYTHONPATH inside collect_data_single.bash. All downstream modules (logger_thread, camera_thread, metadata_utils_parquet, …) pick up the single-arm dimensions and the addverb_single_arm_dataset name without code changes.

    4.1 Docker¶

    docker --version       # 24+ recommended
groups | grep docker   # current user must be in the docker group

    If you are not in the docker group, the run script automatically retries itself under sg docker -c ….

    4.2 SocketCAN¶

    Two CAN interfaces are required:

    Role Interface
    Leader can0
    Follower can1

    Bring them up with the helper that ships with the repo:

    sudo /home/vla-training/isaac_vla/General_Autonomy_Special_Robots/addverb_arms/setup_all_cans.sh

    run_datacollect_docker.sh calls this for you on every launch, so you do not have to remember it.

    4.3 USB cameras¶

    Plug in three V4L2-compatible cameras and run once:

    cd addverb_arms/addverb_vla_datacollection
python3 assign_cameras.py

    This writes /dev/v4l/by-path/... paths into camera_config.json for the three logical roles cam1 (top), cam2 (left), cam3 (right). Persistent by-path entries survive reboots and USB port changes.

    5. First-time build of the container image¶

    The image bakes in:

    • Ubuntu 22.04 (no ROS)
    • liborocos-kdl-dev, libkdl-parser-dev, liburdf-dev, liburdfdom-dev from the Ubuntu universe repo
    • libeigen3-dev, libboost-all-dev, pybind11-dev, build-essentials
    • Python 3.10 + numpy, scipy, pandas, pyarrow, opencv-python, click, pybind11
    • The full addverb_arms/ source tree, with the C++ pybind module compiled at image-build time into python_bindings/pybind/addverb_arms_interface.cpython-310-x86_64-linux-gnu.so
    cd addverb_arms/addverb_vla_datacollection
./run_datacollect_docker.sh                # builds the image if missing
# or, to force a clean rebuild after changing C++ source:
./run_datacollect_docker.sh --rebuild

    Build context is addverb_arms/ (the parent dir). The companion addverb_arms/.dockerignore strips host-side build trees so the context stays small.

    Verify the image and the .so:

    docker images addverb-datacollect:latest

docker run --rm addverb-datacollect:latest \
    bash -lc 'python3 -c "import sys; \
        sys.path.insert(0, \"/workspace/addverb_arms/python_bindings/pybind\"); \
        import addverb_arms_interface as m; \
        print([n for n in dir(m) if not n.startswith(\"_\")])"'
# → ['AddverbArmsJointController', 'ControllerConfig', 'Gain', 'JointState', 'RobotConfig']

    6. Running data collection¶

    6.1 Local-only mode (default)¶

    Everything is saved on the host disk.

    cd addverb_arms/addverb_vla_datacollection
./run_datacollect_docker.sh

    6.2 Remote-streaming mode¶

    Set REMOTE_TRANSFER_ENABLED = True in config_single.py (or pass --remote-host), then start remote_transfer_server.py on the remote PC, then:

    ./run_datacollect_docker.sh --remote-host 192.168.0.56 --remote-port 9200

    The launcher pre-checks reachability and refuses to start the recorder if the remote server is down.

    6.3 The interactive loop¶

    %%{init: {'theme':'base','themeVariables':{'background':'transparent','primaryColor':'#f5f5f5','primaryBorderColor':'#383838','primaryTextColor':'#1f1f1f','lineColor':'#383838','textColor':'#1f1f1f','mainBkg':'#f5f5f5','nodeBkg':'#f5f5f5','nodeBorder':'#383838','clusterBkg':'#fafafa','clusterBorder':'#6b7280','titleColor':'#1f1f1f','edgeLabelBackground':'#ffffff','actorBkg':'#f5f5f5','actorBorder':'#383838','actorTextColor':'#1f1f1f','actorLineColor':'#6b7280','signalColor':'#1f1f1f','signalTextColor':'#1f1f1f','labelBoxBkgColor':'#f5f5f5','labelBoxBorderColor':'#383838','labelTextColor':'#1f1f1f','noteBorderColor':'#B45309','noteBkgColor':'#FEF3C7','noteTextColor':'#78350F'}}}%%
stateDiagram-v2
    [*] --> Ready: container start
    Ready --> Recording: ENTER (start episode)
    Ready --> Ready: r (reset arms only)
    Ready --> [*]: q (quit)
    Recording --> Drained: e or Ctrl+C — drain queue, flush CSV/parquet, home arms
    Drained --> Saved: y (save episode)
    Drained --> Discarded: n (discard)
    Saved --> Decision
    Discarded --> Decision
    Decision --> Ready: y (record another)
    Decision --> [*]: n (done)

    After the container starts you'll see, in order:

    1. READY TO RECORD EPISODE 0 → Press ENTER to start recording (r=reset arms, q=quit):
    2. The recorder spawns: arms home, gains are set, cameras warm up for 3 s, then teleoperation begins. Stats line prints once per second: samples, Hz, skipped, pending (camera queue depth), cam_saved, cam_failures.
    3. To finish the episode, press e (preferred) or Ctrl+C. The recorder drains the camera queue, flushes the local CSV / parquet (or remote stream), then homes both arms.
    4. Save this episode? (y=save / n=discard):
    5. y → CSV + JPEG folders are moved to episodes/episode_NNN/ and the row batch is appended to addverb_single_arm_dataset/data/chunk-000/file-000.parquet; meta/info.json and meta/episodes/... get updated.
    6. n → data.csv and the camera folders are deleted; the remote server is told to discard staged data.
    7. Record another episode? (y/n):

    To abort partway through the loop without recording, press q at the first prompt. r re-homes the arms without starting an episode.

    7. Container privileges (for reference)¶

    run_datacollect_docker.sh launches with everything the recorder needs:

    Flag Why
    --privileged SocketCAN, USB
    --network=host Reach remote_transfer_server over LAN
    --ipc=host Shared-memory frame buffers (/dev/shm)
    --pid=host Cleaner shutdown & camera viewer process tree
    --cap-add NET_ADMIN/NET_RAW/SYS_RAWIO/SYS_ADMIN Bring up CAN, raw frame access
    --device=/dev/videoN Each detected USB camera
    -v /dev/v4l:/dev/v4l:ro Persistent by-path camera names
    -v /dev/bus/usb:/dev/bus/usb All USB devices (cameras, ttyACM)
    -v /tmp/.X11-unix + ~/.Xauthority Optional camera_viewer.py windows on host
    -v <data_collection_dir>:/workspace/.../addverb_vla_datacollection Live edits without rebuild

    The pybind .so is baked into the image under /workspace/addverb_arms/python_bindings/pybind/. We deliberately do NOT bind-mount the pybind directory — that would shadow the built artifact.

    8. Configuration knobs¶

    All of the following live in config_single.py:

    Constant Default Purpose
    LOGGING_RATE_HZ 30.0 Frames per second logged
    CONTROL_LOOP_RATE_HZ 200.0 Leader→follower command loop rate
    CAMERA_WIDTH/HEIGHT 640 × 480 Capture resolution (sensor side)
    LOG_WIDTH/LOG_HEIGHT 320 × 320 Centre-crop + resize before saving / streaming
    REMOTE_TRANSFER_ENABLED False Toggle remote streaming
    REMOTE_HOST/PORT 192.168.0.56:9200 Remote transfer server
    REMOTE_JPEG_QUALITY 90 JPEG quality used for over-the-wire encoding
    DATASET_NAME addverb_single_arm_dataset Output parquet root
    EPISODE_TASK_RANGES [(0, 9999, "Single Arm…", "…")] (start, end, task_name, task_prompt) tuples

    Edit EPISODE_TASK_RANGES to attach different task prompts to different episode-number ranges. The parquet task_index column is resolved from this table at write time.

    9. Troubleshooting¶

    Camera config missing¶

    ERROR: camera_config.json not found!

    You haven't run assign_cameras.py yet. Do it once on the host (it talks directly to V4L2; no docker needed).

    CAN interface not visible inside the container¶

    Confirm on the host first:

    ip link show | grep '^.*can'

    If the interfaces are missing, run setup_all_cans.sh manually and then re-launch the container. --privileged + --network=host is enough for the container to see them — no extra --device flag for can0/1 is required.

    import addverb_arms_interface fails inside the container¶

    The .so wasn't built. Re-build the image with --rebuild:

    ./run_datacollect_docker.sh --rebuild

    Check the build log around the pybind11_add_module step. Most failures fall into:

    • missing apt deps — they should all be in Dockerfile.datacollect; if you've edited it, double-check liborocos-kdl-dev libkdl-parser-dev liburdf-dev liburdfdom-dev liburdfdom-headers-dev libeigen3-dev libboost-all-dev pybind11-dev.
    • addverb_arms_pybind.cpp won't compile after a C++-side change — fix the source then --rebuild.

    Remote server unreachable¶

    ERROR: Remote server at <host>:<port> is NOT reachable.

    Either start remote_transfer_server.py on the remote PC first, or set REMOTE_TRANSFER_ENABLED = False in config_single.py and re-launch.

    Frames being skipped due to camera backlog¶

    The stats line shows pending: [N, N, N] climbing past 10. Either:

    • the host CPU can't keep up encoding 3× JPEGs at 30 Hz — drop LOG_WIDTH/HEIGHT or LOGGING_RATE_HZ in config_single.py,
    • or remote streaming is saturated. Check [RemoteClient] queue depth printout — if it stays high, drop REMOTE_JPEG_QUALITY (e.g. 90 → 75).

    Can I run dual-arm and single-arm from the same checkout?¶

    Yes — none of the patches change default behaviour.

    • Dual-arm: bash collect_data.bash
    • Single-arm: bash collect_data_single.bash (or via the docker launcher)

    10. Quick reference¶

    # Build / launch container (does both)
./addverb_arms/addverb_vla_datacollection/run_datacollect_docker.sh

# Force rebuild image after C++ source changes
./addverb_arms/addverb_vla_datacollection/run_datacollect_docker.sh --rebuild

# Stream to a remote PC
./addverb_arms/addverb_vla_datacollection/run_datacollect_docker.sh \
    --remote-host 192.168.0.56 --remote-port 9200

# Run on bare host (no container) — same flags, same output
cd addverb_arms/addverb_vla_datacollection
bash collect_data_single.bash
