    Sim-to-Real Training Pipeline (DR, GR00T, Co-training)¶

    End-to-end recipe for training a vision-language-action (VLA) policy on top of the Addverb teleoperation stack and deploying it back onto the physical arm. Targets our hardware (7-DoF Addverb arm — 6 active revolute joints plus a welded wrist flange in sim + 1 prismatic gripper) and our dataset (2 USB cameras for single-arm captures, LeRobot Parquet/JPEG via addverb_vla_datacollection).

    The recipe has five stages:

    The choice between DR-only (Stages 1–4) and co-training (adds Stage 5) is empirical: try DR-only first; only fall through to co-training if real-world success rate is unacceptable.

    Stage 1 — Domain-Randomised Teleoperation Dataset¶

    Why DR¶

    Real cameras drift, lighting changes, mounts vibrate. A policy trained on a single, perfectly-lit sim scene will overfit to those exact conditions and fail on real hardware. We randomise the sim scene at every reset so the policy can never lean on visual constants — it has to learn the task, not the backdrop.

    The Addverb env already wires DR into addverb_isaaclab/sim_agent/mdp/resets.py via three event terms in addverb_env_cfg.py:

    Helper Purpose YAML knob (under domain_randomization:)
    randomize_camera_focal_length Perturb FOV of a camera sensor camera_ego_focal_length.{enabled, range_mm}
    randomize_camera_pose Perturb camera position / orientation camera_external_pose.{enabled, pos_range, rot_range}
    randomize_light_exposure Vary scene brightness light_exposure.{enabled, range_ev}
    random_asset_pose Randomise cube/asset pose always-on; driven by cube.spawn_range

    Each block has its own enabled: true|false flag (default false); a disabled block adds no EventTermCfg, so the env behaves identically to a non-DR run. Toggle individual knobs without rebuilding anything.

    The current YAML in addverb_isaaclab/config/isaac_scene_config.yaml (domain_randomization: block) is:

    domain_randomization:
  camera_ego_focal_length:
    enabled: false
    range_mm: [12.0, 15.0]              # mm  (±10 % around 13.5)

  camera_external_pose:
    enabled: false
    pos_range:                          # m, per-axis delta
      x: [-0.02, 0.02]
      y: [-0.02, 0.02]
      z: [-0.01, 0.01]
    rot_range:                          # rad, per-axis delta
      roll:  [-0.05, 0.05]
      pitch: [-0.05, 0.05]
      yaw:   [-0.05, 0.05]

  light_exposure:
    enabled: false
    range_ev: [-3.0, 1.0]

    Flip the relevant enabled: true to activate a knob — no other change is required.

    Recommended ranges¶

    Knob Recommended range Status in our config
    Light exposure (-4.0, 3.0) We use (-3.0, 1.0) (slightly narrower — widen if real lighting varies more)
    Light colour temp. (2500.0, 9500.0) K Not yet wired — add a randomize_light_colour_temp helper if HDRI lighting drifts in your real scene
    HDRI texture procedurally selected from available .exr files Not yet wired — irrelevant if our real workspace is fixed
    Camera position X / Y (-0.02, 0.02) m Same
    Camera position Z (-0.01, 0.01) m Same
    Roll / pitch (-0.05, 0.05) rad Same
    Yaw (-0.1, 0.1) rad We use (-0.05, 0.05) — widen if external camera mount is hand-tightened in real
    Object pose randomised per reset Yes — reset_cube_position event term

    Physics randomisation categories (object mass, friction, restitution, joint damping, actuator delays, sensor noise) have no fixed numeric ranges; treat that as an open knob if your real arm's torque profile diverges from the URDF.

    Episode count¶

    Plan for 70–150 successful takes per task prompt. Below 50 the policy underfits; beyond 200 the marginal returns flatten.

    Recording with the data-collection container¶

    For the real-hardware single-arm path the workflow is the run_datacollect_docker.sh loop documented in Real-Hardware Data Collection. For the sim teleop path you publish leader joint frames over ZMQ (addverb_isaaclab/bridge/bridge_sender.py) and the in-container agent (addverb_agent.py) writes the episode.

    Step Real-hardware (this repo)
    Start episode ENTER at the master-script prompt
    Stop episode press e (or Ctrl+C)
    Cancel mid-take press q at the next prompt + n to discard
    Reset arm only r at the master-script prompt

    Task prompt strings¶

    Pin one English sentence per task and store it as task_prompt. The location depends on which capture path you use:

    • Sim capture (addverb_isaaclab): addverb_isaaclab/config/dataset_config.yaml → task_ranges. Each entry is [start_ep, end_ep, task_name, task_prompt]. SimDataLogger._get_task_prompt(episode_num) resolves the active prompt at log time and stamps it into both each row and meta/tasks.parquet.
    • Real-hardware capture (addverb_vla_datacollection): EPISODE_TASK_RANGES in config_single.py (or config.py for the dual-arm setup).

    Update the task_prompt string before recording so the parquet task_index column points at the right description. The default single-arm prompt is:

    "Use the leader arm to teleoperate the follower arm and complete the
 manipulation task."

    Replace this with the actual task before each recording session — GR00T conditions on it at inference time.

    Dataset format produced¶

    Per frame, written into the LeRobot v2 layout described in Real-Hardware Data Collection:

    • observation.state.joint_positions — float32 [7] (6 arm joints + gripper)
    • action.robot — float32 [7] (leader joints + gripper)
    • observation.images.camera1_images — JPEG (top)
    • observation.images.camera2_images — JPEG (left)
    • observation.images.camera3_images — JPEG (right)
    • task_index, episode_index, frame_index, timestamp

    Our three camera keys (camera1_images, camera2_images, camera3_images) need a rename map at training time to whatever canonical names the trainer expects (e.g. top / left / right) — see Stage 2.

    Camera framing. Default is 320×320 (square). If you change LOG_WIDTH/LOG_HEIGHT in config_single.py, keep the aspect ratio sensible.

    Stage 2 — Fine-tuning GR00T on the Dataset¶

    Base model¶

    nvidia/GR00T-N1.6-3B is the base checkpoint.

    Training command¶

    CUDA_VISIBLE_DEVICES=0 python \
    gr00t/experiment/launch_finetune.py \
    --base_model_path nvidia/GR00T-N1.6-3B \
    --dataset_path $DATASET_PATH \
    --modality_config_path examples/AddverbArm/addverb_config.py \
    --embodiment_tag NEW_EMBODIMENT \
    --num_gpus $NUM_GPUS \
    --output_dir /tmp/addverb_finetune \
    --save_steps 1000 \
    --save_total_limit 5 \
    --max_steps 10000 \
    --warmup_ratio 0.05 \
    --weight_decay 1e-5 \
    --learning_rate 1e-4 \
    --use_wandb \
    --global_batch_size 32 \
    --color_jitter_params brightness 0.3 contrast 0.4 saturation 0.5 hue 0.08 \
    --dataloader_num_workers 4

    Hyper-parameters¶

    Parameter Value
    Learning rate 1e-4
    Global batch size 32
    Max training steps 10 000
    Warmup ratio 0.05
    Weight decay 1e-5
    Action horizon 16
    Save every 1 000 steps

    Hardware¶

    Resource Requirement
    GPU 24 GB+ VRAM
    Training time 2–8 h (dataset-dependent)
    Inference target RTX 3080+ (real-time)

    Data requirements¶

    50–200 demonstrations per task for basic competence, with language annotations and varied conditions.

    Required configuration¶

    1. Dataset path. Point --dataset_path at the LeRobot root produced by Stage 1:
    export DATASET_PATH=/home/vla-training/isaac_vla/General_Autonomy_Special_Robots/\
addverb_arms/addverb_vla_datacollection/addverb_single_arm_dataset

    1. Modality config. Author examples/AddverbArm/addverb_config.py describing:

    2. Joint count → 7 (6 active arm revolutes + 1 prismatic gripper; link_7 is welded as a fixed flange in sim and never enters the vector).

    3. Camera-name mapping that our parquet produces depends on the capture path: single-arm hardware emits camera1_images, camera2_images (top + wrist); the sim follower emits observation.images.ego and observation.images.external. Pick canonical names GR00T expects (e.g. top, wrist, or ego/external) and apply the rename via the config or via --rename_map at train time.

    4. Embodiment tag. --embodiment_tag NEW_EMBODIMENT is fine for a first run — GR00T will treat the Addverb arm as an unseen embodiment and fine-tune accordingly. Once you have a stable checkpoint, give it a stable tag (e.g. ADDVERB_7DOF) so co-trained datasets line up.

    Real-arm client (Terminal 2, on the data-collection PC)¶

    Author addverb_eval.py under Isaac-GR00T/gr00t/eval/real_robot/Addverb/. Concretely it must:

    1. Drive the follower over its CAN interface (reuse the AddverbArmsJointController Python binding from addverb_arms/python_bindings/pybind/).
    2. Open three OpenCV cameras using the V4L2 paths from camera_config.json, capturing at 640×480 @ 30 fps: 
      cameras = {
    "top":   {"index_or_path": cam1_path, "width": 640, "height": 480, "fps": 30},
    "left":  {"index_or_path": cam2_path, "width": 640, "height": 480, "fps": 30},
    "right": {"index_or_path": cam3_path, "width": 640, "height": 480, "fps": 30},
}
    3. Connect to the GR00T server at --policy_host localhost --policy_port 5555.
    4. Pass --lang_instruction matching the trained prompt.
    5. Stream actions back to the arm at the same Hz the dataset was logged at (we use 30 Hz logging, while the underlying control loop runs at CONTROL_LOOP_RATE_HZ = 200 Hz; GR00T's action_horizon=16 chunk gives a smooth interpolation buffer).

    Success metrics¶

    There is no automated success scorer in the eval pipeline. Plan to:

    • Run at least 20–30 rollouts per task so you can quote success rate with a useful confidence interval.
    • Have a human note success / failure / safety_intervention on each rollout in a CSV.

    Safety¶

    The bridge already homes both arms on graceful shutdown (see _shutdown_sequence() in addverb_arms/addverb_vla_datacollection/data-collect.py). Reuse the same logic in addverb_eval.py. Add explicitly:

    • A workspace box check that rejects any commanded pose outside the table volume.
    • A physical e-stop wired to the leader controller.

    Stage 5 — Co-Training¶

    When real-only fine-tuning is unaffordable (we cannot collect 200 real demos per task) and sim-only DR has a residual sim-to-real gap, mix the two corpora.

    The principle¶

    Source Quantity Quality Reality match
    Simulation Abundant Consistent Approximate
    Real teleop Limited Variable Exact

    Each source plugs the other's hole.

    Recommended mixing ratio¶

    Roughly 14 : 1 to 20 : 1 sim : real — a small set of real takes (≈5 episodes) paired with a much larger sim set (70–100). Concretely, for our Addverb pick-and-place:

    • Real: collect 5–10 good takes via run_datacollect_docker.sh (Stage 1, real path).
    • Sim: collect 70–100 takes with full DR via addverb_agent.py (Stage 1, sim path with DR turned on).

    Apply the same DR ranges from §Stage 1 to the sim half — that is the whole point.

    The default approach for Addverb:

    1. Produce two LeRobot dataset roots:
    2. addverb_real_dataset/ (Stage 1 real path)
    3. addverb_sim_dataset/ (Stage 1 sim path, full DR)
    4. Concatenate at the parquet level into a third root addverb_cotrain_dataset/ with episode_index re-numbered so that the real episodes appear first (operator can spot-check them easily) and sim episodes second.
    5. Train with the same Stage 2 GR00T command, pointing --dataset_path at addverb_cotrain_dataset/.
