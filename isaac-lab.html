    Addverb Hybrid Isaac Lab Simulator¶

    This directory contains the necessary components to run an Isaac Lab (Isaac Sim 5.1) simulation of the Addverb Arm inside a Docker container, seamlessly controlled by hardware or mock data running on the host via ZMQ.

    Architecture¶

    To avoid dependency conflicts between ROS, the lerobot Python framework, and Isaac Sim's internal frozen Python environment, we use a hybrid architecture:

    1. Isaac Lab Simulation (Container): Runs natively inside the NVIDIA base image (nvcr.io/nvidia/isaac-lab:2.3.2). Contains zero dependencies on lerobot_framework.
    2. Hardware / Mock Bridge (Host): Reads the real arm state via the Pybind C++ SDK (or generates mock sine-wave data) and publishes it via a highly efficient ZeroMQ (tcp://0.0.0.0:5556) PUB/SUB topic.
    3. Data Logger (Container): Mirrored from the physical hardware (metadata_utils_parquet.py). Automatically saves simulation observations, states, and camera images into the exact Parquet directory structure used by the hardware setup.

    1. Zero-Configuration Build¶

    Build the addverb-isaaclab:latest container:

    cd /home/vla-training/isaac_vla/General_Autonomy_Special_Robots
./docker/build_sim.sh

    The robot USD lives in the repo at addverb_isaaclab/assets/usd/addverb_arm/ and is mounted into the container at runtime. No URDF→USD conversion happens at launch.

    2. Robot Asset¶

    The pre-converted robot USD lives at: 

    addverb_isaaclab/assets/usd/addverb_arm/addverb_gripper_urdf_2.usd
    with sub-USDs in configuration/ (base + physics + sensor). run_sim.sh errors out early if this file is missing. Replace the bundle in place if you need to update the robot.

    A pair of launcher scripts reads launch_config.yaml at the repo root and drives everything from one file:

    # Terminal 1 — mock (or real) bridge on the HOST
./scripts/launch_bridge.sh

# Terminal 2 — containerised sim
./scripts/launch_sim.sh

    Switch mode: mock ↔ mode: real in launch_config.yaml to toggle between mock_bridge_sender.py and the real hardware bridge. Per-run overrides: EPISODE_NUM=3 ./scripts/launch_sim.sh, MODE=real ./scripts/launch_bridge.sh.

    Manual pipeline (without launchers)¶

    Mock sine-wave test (host): 

    conda activate unified_addverb_isaaclab
python addverb_isaaclab/bridge/mock_bridge_sender.py --sine

    Real arm (host, CAN bus): 

    conda activate unified_addverb_isaaclab
python addverb_isaaclab/bridge/hardware_bridge_sender.py --interface can0 --model ADD

    Sim (container, separate terminal): 

    ./docker/run_sim.sh --task Addverb-Teleop-Cam-v0

    The graphical viewport opens via X11 forwarding. The arm tracks whatever the bridge publishes.

    4. Collecting Data¶

    Pass --repo_root + --episode_num to run_sim.sh and the SimDataLogger writes the same parquet + JPEG layout the real-robot pipeline uses:

    ./docker/run_sim.sh --task Addverb-Teleop-Cam-v0 \
    --repo_root /workspace/datasets/session_001 --episode_num 0

    You'll get a datasets/session_001/ tree with meta/info.json, meta/tasks.parquet, data/chunk-000/file-000.parquet, and per-camera images/observation.images.<cam>/episode_NNN/*.jpg.
